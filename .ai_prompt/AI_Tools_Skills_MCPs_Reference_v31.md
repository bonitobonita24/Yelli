## 1. Agent Stack

Six agents form the complete automation layer. Each has a hard phase boundary (Rule H4).

| Agent | Phase Boundary | What It Does | How It Loads |
|---|---|---|---|
| **Claude Code** | ALL phases (V31 primary) | Primary agent. **Opus 4.6 = Architect** (planning, decomposition, review) and **fallback Executor** for tasks that are genuinely atomic but exceed Sonnet's 30K token budget (max 20% of tasks, 100K budget). **Sonnet 4.6 = Executor** (implementation, tests, commits via subagents — each task must estimate ≤30K tokens). See `memory-governance.md` Section 4 for the Architect-Execute Model and Step 2.5/2.5b for the 30K Budget Gate. Handles Bootstrap through Phase 8, Feature Updates, Resume, Gov Sync | Auto-loads compact CLAUDE.md (~200 lines). Reads `.claude/rules/` contextually per task. Reads `memory-governance.md` at every phase pre-flight |
| **Cline** | ⚠ DEPRECATED (V31) — do not use | .clinerules still generated by Bootstrap but unused. Claude Code handles all work Cline used to handle. Kept in framework for historical reference only. | N/A — deprecated |
| **Copilot** | Inline autocomplete + PR review | Always-on autocomplete while you type. Fallback if primary agent hits unresolvable error | VS Code extension — always on |
| **SpecStory** | Passive (all phases) | Captures every Claude Code session to `.specstory/history/`. Powers Governance Sync attribution | VS Code extension — zero config |
| **SocratiCode** | Phase 4 Part 8 onward (after scaffold) | Semantic codebase search. 61% less context, 84% fewer tool calls, 37x faster than grep | MCP server via npx — Docker required |
| **code-review-graph** | Phase 7 (Feature Updates) | Structural blast-radius: what breaks if I change this? Tree-sitter graph + SQLite | Claude Code plugin — one-time per machine |

> **Note — Planning Assistant (NEW V31 role expansion):**
> The Planning Assistant chat runs on **claude.ai** (external to the project) using `Product_md_Planning_Assistant_v31.md` as its system prompt. It produces PRODUCT.md during Phase 2 interview and generates an ephemeral HTML clickable mockup during Phase 2.8 (V31). It is NOT counted as one of the 6 in-project agents because it never reads or writes any project file — the human copies PRODUCT.md into the project after the Planning Assistant chat concludes.

## 2. MCP Servers

MCP servers run alongside the agents as persistent local services. They extend what agents can do.

### 2.1 SocratiCode
```
Repo:     (runs via npx socraticode)
Type:     MCP server — semantic codebase intelligence
Added:    V10
Install:  Runs automatically via .vscode/mcp.json entry (Bootstrap Step 10)
          Requires: Docker Desktop running on Windows (WSL2 accesses natively)
```

**What it does:** Hybrid semantic + keyword search across the entire codebase. Builds an AST-aware chunk index using Qdrant (vector DB) + Ollama (embeddings), both running in Docker. On first use it auto-pulls both images (~5 min one-time).

**21 tools including:**
- `codebase_search` — semantic search for concepts ("where is auth handled?")
- `codebase_context_search` — retrieves pre-built context artifacts (schema, docs)
- `codebase_index` — build or rebuild the full index
- `codebase_update` — incremental update after file changes
- `polyglot_dependency_graph` — cross-language dependency map

**When agents use it:**
- Phase 4 Part 8: initial index built after scaffold completes
- Phase 7 Step 2: `codebase_search` before touching any file (Rule 17)
- Phase 7 Step 10: `codebase_update` after implementation

**`vscode/mcp.json` entry (Bootstrap Step 10 generates this):**
```json
{
  "servers": {
    "socraticode": {
      "command": "npx",
      "args": ["-y", "socraticode"]
    }
  }
}
```

### 2.2 code-review-graph
```
Repo:     github.com/tirth8205/code-review-graph
Type:     MCP server — structural blast-radius analysis
Added:    V13
Install:  claude plugin add tirth8205/code-review-graph  (one-time per machine)
          Requires: Python 3.10+, uv package manager
```

**What it does:** Builds a persistent Tree-sitter graph of the codebase stored in SQLite with NetworkX traversal. Answers "what breaks if I change this?" — structural impact, not semantic search. Supports 12 languages.

**8 tools including:**
- `get_impact_radius_tool` — blast radius: what files/functions break if this changes
- `get_review_context_tool` — token-optimized context bundle (changed files + impacted)
- `query_graph_tool` — callers, callees, tests, imports for any symbol
- `build_or_update_graph_tool` — full or incremental graph build
- `semantic_search_nodes_tool` — search by name or semantic similarity

**When agents use it:**
- Phase 7 Step 3: `get_impact_radius_tool` before every Feature Update (Phase 7 blast-radius check)
- Two-stage review (Rule 25): `get_review_context_tool` for scope-limited review

**Per-project setup (after machine install):**
```bash
# Run once per project after Phase 4 completes
code-review-graph build
```

**How it complements SocratiCode:**
| Question | Tool |
|---|---|
| "Where is auth implemented?" | SocratiCode `codebase_search` |
| "What breaks if I change this entity?" | code-review-graph `get_impact_radius_tool` |
| "What tests cover this function?" | code-review-graph `query_graph_tool` |
| "Show me the schema structure" | SocratiCode `codebase_context_search` |

```
Type:     Framework skill — always scaffolded
```

### 3.8 Memory Governance Layer (V31.2)
```
Type:     Governance rules file (not a skill — a .claude/rules/ module)
Added:    V31.1
Location: .claude/rules/memory-governance.md
Install:  Auto-installed by /scan-project on Spec-Driven projects
          Manual: copy from docs/SpecDrivenAIMegaPrompt/memory-governance.md
```

**What it does:** Prevents context thrashing via 5 integrated systems:
- **Section 1 — Tiered Decomposition:** 3-tier complexity classifier with scoring formula. Runs at pre-flight.
- **Section 2 — Smart Checkpoint:** Auto-persists progress to STATE.md + Claude Code memory. Zero-cost resume.
- **Section 3 — Phase Hooks:** One-liner hooks in 12 phase pre-flights enforce governance.
- **Section 4 — Architect-Execute Model:** Opus 4.6 plans, dispatches Sonnet 4.6 subagents via `Agent(model: "sonnet")`. Uses `superpowers:subagent-driven-development` for orchestration.
- **Section 5 — Mid-Project Adoption:** 4-step path + thrashing recovery for Phase 7/8 projects.
- **Step 2.5 — 30K Token Budget Gate:** Every Sonnet subagent task must have a token estimate ≤30K before dispatch. If over, split further regardless of tier.
- **Step 2.5b — Opus Escalation:** If a task is genuinely atomic and unsplittable but >30K, dispatch to `Agent(model: "opus")` as last resort (100K budget, max 20% of tasks).
- **THRASHING Status:** New Sonnet status detected by Opus. Response: stop immediately, re-decompose.

**When it activates:**
- Every phase pre-flight (Tiered Decomposition)
- Every task completion that modified files (Smart Checkpoint)
- Phase 4/7/8 work (Architect-Execute Model)
- Every subagent dispatch (30K Budget Gate check)

**How it connects to PRODUCT.md:**
- Opus reads PRODUCT.md sections during planning; Sonnet never reads PRODUCT.md directly
- Decomposition decisions are based on PRODUCT.md feature scope
- Smart Checkpoint captures decisions that came from PRODUCT.md interpretation

### 4.4 Xendit (payment gateway — framework default)

```
Type:     External SaaS — conditional (only if app accepts payments)
Added:    V27
Phase:    Phase 2 Section G2 → PRODUCT.md. Bootstrap Step 18 Section 4.5 → CREDENTIALS.md.
```

Framework default payment gateway for all SEA markets (ID, PH, MY, TH, VN). Accepts cards, e-wallets, bank transfers, OTC/over-the-counter, and QR payments. Webhook verification via `x-callback-token` header (constant-time comparison). API uses Basic Auth with secret key as username.

**PRODUCT.md signal:** If any module or workflow mentions payment acceptance, Phase 2 Section G2 asks payment questions and locks `payment.gateway: xendit` in DECISIONS_LOG.md.

**Credentials:** Bootstrap Step 18 Section 4.5 collects 5 values: test secret key, live secret key, test public key, live public key, webhook verification token. Stored in dedicated CREDENTIALS.md Xendit section. Written to `.env.dev` (TEST keys), `.env.staging/.env.prod` (LIVE keys). All gitignored.

**Security:** Secure Code Generation section enforces 6 rules: x-callback-token verification, idempotent webhook handlers, payload validation, endpoint security, key isolation, queue processing recommendation.

**Docs:** https://docs.xendit.co/apidocs · **Webhooks:** https://docs.xendit.co/docs/handling-webhooks · **Dashboard:** https://dashboard.xendit.co

### 4.5 Cloudflare Turnstile (bot protection — framework default)

```
Type:     External SaaS — framework default (enabled unless opted out)
Added:    V27
Phase:    Phase 2 Section H → PRODUCT.md. Bootstrap Step 18 Section 4.6 → CREDENTIALS.md.
```

Framework default bot protection replacing CAPTCHA. Invisible or managed challenges — visitors rarely see anything. WCAG 2.2 AAA compliant. FREE tier: 20 widgets (1 per app sufficient), 10 hostnames per widget (only prod domain needed — dev + staging use test keys = 1 hostname per app).

**Protected pages by default:** login, registration, password reset, contact forms, payment pages (if Xendit enabled). Authenticated pages NOT protected (rate limiting suffices).

**Credentials:** Bootstrap Step 18 Section 4.6 collects Site Key (public, client-safe) + Secret Key (server-only). Dev AND staging use Cloudflare's official test keys (always-pass) — zero real widget needed except in prod. Stored in CREDENTIALS.md Turnstile section + .env files. All gitignored.

**Integration:** Client-side `@marsidev/react-turnstile` React component (explicit rendering). Server-side `siteverify` POST validation mandatory — token is single-use, expires in 300s. CSP must allow `challenges.cloudflare.com`.

**Multi-tenant SaaS:** Each unique hostname uses 1 of 10 slots. Subdomains of your domain can use wildcard. Custom tenant domains need individual hostname entries. Phase 2 Section H asks about hostname count.

**Docs:** https://developers.cloudflare.com/turnstile/ · **Testing:** https://developers.cloudflare.com/turnstile/troubleshooting/testing/

## 5. Model Routing

Locked in `inputs.yml` during Phase 3. Every project gets the same default routing unless explicitly changed in PRODUCT.md or DECISIONS_LOG.md.

```yaml
models:
  planning:   claude-code           # Phase 2 interview — CLAUDE.md auto-loaded
  execution:  claude-sonnet-4-6     # Phase 4-8, Feature Updates — V31 primary via Claude Code
  governance: gemini-2.5-flash-lite # Changelog/doc writes — cheapest
  escalation: claude-opus-4-6       # Complex multi-file bugs — when Sonnet 4.6 pivots
```

### Why this routing

| Model | Role | Reason |
|---|---|---|
| **Claude Code** | Planning | Auto-loads CLAUDE.md. Best for spec interview, ambiguity resolution, Phase 2 reasoning |
| **Claude Sonnet 4.6** | Execution (V31 primary) | Promoted to primary in V30. Handles ALL Phase 4-8 scaffold and Feature Updates via Claude Code. Replaces MiniMax M2.5 which was used via Cline (Cline deprecated V31). |
| **Gemini 2.5 Flash-Lite** | Governance | Cheapest capable model. Governance writes (CHANGELOG_AI, DECISIONS_LOG, IMPLEMENTATION_MAP) are low-complexity document updates |
| **Claude Opus 4.6** | Escalation | Invoked only when Sonnet 4.6 hits 2-strike failure on complex multi-file reasoning. Not the default — escalation only |

### Legacy MiniMax M2.5 routing (historical — V19 through V30)

Before V31 Cline deprecation, MiniMax M2.5 was the default execution model on OpenRouter free tier because Cline ran execution via OpenRouter. V31 in-place deprecation moved execution to Claude Code + Claude Sonnet 4.6 directly. MiniMax references below are preserved as historical context only:

- V19-V30 default execution: `minimax/minimax-m2.5` on OpenRouter free tier
- V19-V30 OpenRouter troubleshooting: if `404 No endpoints available`, relax privacy settings at https://openrouter.ai/settings/privacy or fallback to `deepseek/deepseek-v3`
- V19-V30 execution fallback: `deepseek/deepseek-v3` (comparable SWE-Bench score, free tier, strong at spec decomposition)
- V31+: execution routes through Claude Code; MiniMax/DeepSeek only apply if user manually re-enables Cline as emergency fallback (not part of framework routing)

This is the core of intelligent app building — every tool decision traces back to what's declared in PRODUCT.md.

### 6.1 The full decision chain

```

─────────────────────────────────────────────────────────────────────

  Marine Conservation

  banking                   (not shared schema — override of default)

  offline-first: yes        Mobile: React Native Reusables + NativeWind
  platform: iOS/Android     Push notifications: Expo Push wired
accessibility: wcag_aa    → a11y skill activated (blocking pre-delivery)
Design Identity present   → UI UX Pro Max runs in Phase 2.6
  (Section K)               Generates design-system/MASTER.md
Background jobs declared  → BullMQ + Valkey worker scaffolded
  (BullMQ: yes)             docker-compose.cache.yml + worker service

  (storage: minio)          presigned URL helper generated

  (realtime: sse)           tRPC + React Query polling (mobile)

  (shared schema + RLS)     RLS policies enabled in Prisma schema

                              Zero-migration upgrade path scaffolded

                              COMMANDS.md docker push section

```

### 6.2 Phase-by-phase tool activation

```
Phase 0  Bootstrap
  → Writes .vscode/mcp.json (SocratiCode + Context7)
  → Bootstrap Step 17: spec-driven-core skill + prompts for frontend-design, a11y
  → Bootstrap Step 18: CREDENTIALS.md written (GitHub, Docker Hub, SMTP, Komodo, Xendit, Turnstile)

Phase 2  Discovery Interview (Claude Code)
  → CREDENTIALS.md pre-flight gate
  → Reads PRODUCT.md: locks tenancy, entities, security, infrastructure
  → Locks model routing in inputs.yml

Phase 2.6  Design System (Claude Code, conditional — Cline deprecated)
  → ONLY if UI UX Pro Max skill installed AND Design Identity in PRODUCT.md
  → Reads Industry + Design Identity → generates design-system/MASTER.md
  → Embeds Vercel Web Interface Guidelines in MASTER.md
  → Detects a11y requirement from PRODUCT.md → flags for enforcement
  → Detects shadcnblocks skill → adds block catalog reference to MASTER.md
  → Detects frontend-design plugin → notes it for Phase 4/7 UI tasks

Phase 2.7  Spec Stress-Test (Claude Code, auto)
  → Pressure-tests PRODUCT.md for gaps before any code is written
  → Blocks Phase 3 if required sections are missing or contradictory

Phase 3  Generate Spec Files (Claude Code)
```

## 9. Governance Tools (tools/ directory)

Four `.mjs` files generated by Phase 4 Part 7. These are the only `.mjs` files in the project (the one exception to Rule 12 TypeScript-everywhere — they must be Node scripts for CI compatibility).

### 9.1 validate-inputs.mjs
Validates `inputs.yml` against `inputs.schema.json`. Runs in CI (`pnpm tools:validate-inputs`) and Phase 5. Exits 1 if any required field is missing or wrong type. Prevents agents from building on a broken spec.

### 9.2 check-env.mjs
Validates that every env var declared in `inputs.yml` exists in the current `.env.*` file and is non-empty. Runs at Phase 5 and CI. Catches missing secrets before deployment. No more "undefined DATABASE_URL" runtime crashes.

### 9.3 hydration-lint.mjs
**This is the tool that prevents hydration errors.** Scans the Next.js `apps/[web]/src/` directory for known hydration anti-patterns:

| Pattern detected | Why it causes hydration errors |
|---|---|
| `typeof window !== 'undefined'` in render | Server/client HTML mismatch |
| `Math.random()` or `Date.now()` in render | Non-deterministic render output |
| `localStorage` / `sessionStorage` in render | Not available server-side |
| `new Date()` without timezone normalization | Server UTC vs client local time |
| Dynamic class names without `cn()` | Class order differs server/client |
| `useLayoutEffect` without SSR guard | Runs on server, throws warning |
| Missing `suppressHydrationWarning` on time elements | Known-safe dynamic timestamps |

Runs as part of Phase 5 validation and CI. Exits 1 on any violation. Agents must fix before merge.

### 9.4 check-product-sync.mjs
Implements Rule 9 (bidirectional governance). Checks:
- Direction A: Every entity in `PRODUCT.md` has a corresponding Prisma model
- Direction B: Every Prisma model is declared in `PRODUCT.md`
- Every module declared in `PRODUCT.md` has a corresponding tRPC router
- `<private>` tags in `PRODUCT.md` have not leaked into governance docs

Exits 1 on any violation. Runs in CI and Phase 5. This is the primary guard against spec drift.

## 10. Security Stack (always scaffolded — V18+)

These four security primitives are generated in every app regardless of PRODUCT.md. They are not optional.

### 10.1 HTTP Security Headers
Generated in `next.config.ts` for every Next.js app:

```ts
  { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains; preload' },

  { key: 'Content-Security-Policy', value: "default-src 'self'; ..." },
```

### 10.2 LRU Rate Limiter
Generated in `src/server/lib/rate-limit.ts`. Four pre-configured tiers:

| Tier | Limit | Use |
|---|---|---|

## 13. TypeScript Strictness Config

Every project uses this exact `tsconfig.base.json`. All flags are required. None are optional.

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  }
}
```

ESLint rules (always enforced):
- `@typescript-eslint/no-explicit-any: error` — no `any` types ever
- `@typescript-eslint/no-unsafe-assignment: error` — no implicit `any` from untyped
- `@typescript-eslint/strict-boolean-expressions: error` — no truthy coercion

**These strictness flags are why hydration-lint exists.** `noUncheckedIndexedAccess` and `exactOptionalPropertyTypes` catch entire classes of runtime errors at compile time — but they also surface edge cases that cause hydration mismatches. The lint tool catches what TypeScript can't see.

## 15. Secure Code Generation (V25+)

This is not a tool or skill — it is a mandatory instruction block that Claude Sonnet 4.6 via Claude Code (V31 primary) and all other agents must follow during Phase 4 scaffold and Phase 7 Feature Updates.

**16 sub-sections, 15+ threat scenarios addressed.** Key constraints:
- L6 Prisma extension uses `$allOperations` — covers delete, count, aggregate (previously unguarded)

- CSRF protection documented (tRPC + SameSite inherently resistant)

- Session invalidation on role/tenant change
- Tiered global rate limiting (auth/API/public)
